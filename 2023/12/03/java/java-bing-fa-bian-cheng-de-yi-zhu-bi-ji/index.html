JAVA 并发编程的艺术

JUC NOTE
JAVA
发布日期:   2023-12-03

第一章 并发编程的挑战

1.1上下文切换

CPU通过时间皮分配算法进行任务执行 档当前任务执行一个时间片后 会切换到下一个任务 在切换之前 会保存当前的任务状态 以便下次切换回当前任务时 能够加载任务状态 所以任务从保存到加载的过程就是一次上下文切换

1.1.1 多线程并不一定更快

1.如果计算数据量小 线程的创建和上下文切换的开销比数据操作更加花费时间。

2.资源限制 如 网络带宽 CPU资源不足。

1.1.2 减少上下文切换

  1. 无锁并发编程 (不同线程处理不同数据段 减少多线程争夺锁的过程)
  2. CAS算法 (用CAS来更新数据 不用加锁)
  3. 合理创建线程 (避免创建不需要的线程 让大量线程等待)
  4. 协程 (在单线程中维持多个任务的切换)

1.2 死锁

1.2.1 什么是死锁

死锁是在多个进程或线程之间互相等待对方释放资源而无法继续执行的一种情况。这种情况发生在每个进程都在等待另一个进程所持有的资源，导致所有进程都无法向前推进，进而陷入了僵局

1.2.2 如何避免死锁

  1. 避免一个线程同时获取多个锁
  2. 避免一个线程在锁内占有多个资源
  3. 尝试使用定时锁

第二章 java 并发机制的底层实现原理

2.1volatile

volatile 是轻量级的synchronization 它在多处理器开发中保证了共享变量的可见性
如果一个字段被声明成volatile java 线程内存模型确保所有的线程看到这个变量值是一致的

2.1.1 相关的CPU术语

内存屏障 是一组处理器指令 用于实现对内存操作的顺序限制
缓冲行 CPU高速缓冲中可以分配的最小存贮单位 处理器会加载整个缓存行
原子操作 不可中断的一个或者一系列操作
缓存行填充 当处理器识别到从内存中读取操作是可缓存的 处理器读取整个高速缓存行到适当的缓存

2.1.2 volatile 是如何保证可见性的

在 volatile 修饰的共享变量中 汇编指令会加上lock 前缀 在lock 命令下 会发生一下两件事情

  1. 将当前处理器缓存行的数据写回到系统内存
  2. 这个写回内存的操作会使在其他CPU的缓存了该内存地址的数据无效
    PS: 为了提高处理速度 处理器不直接和内存进行通信 而是先将系统内存的数据读到内部缓存(L1,L2,L3) 后在进行操作 JVM就会向处理器发送一条LOCK前缀指令 将这个变量所在缓存行的数据写回到系统内存 但是就算是写回到系统内存 如果其他处理器缓存的值还是旧的
    在执行还是会有问题 所以 在多处理器下 为了保证各个处理器的缓存是一致的 就会实现缓存一致性协议 每个处理器通过嗅探在总线上传播的数据来检查自己的缓存是不是过期了
    当处理器发现自己缓存行对应的内存地址被修改 就会将当前处理器的缓存行设置成为无效状态 当处理器需要对这个数据进行修改操作的时候 会重新从系统内存中把数据读到处理器的缓存里

2.1.3 举例

2.1.3.1 内存可见性

在这个例子中，一个线程写入了一个 volatile 变量 flag，另一个线程不断检查 flag 是否变为 true。volatile 保证了当写线程修改了 flag 的值后，读线程能够立即看到这个变化，确保了内存可见性。因此，当写线程将 flag 置为 true 时，读线程能够立即退出循环并打印相应的消息，这展示了内存可见性的效果。

2.1.3.2 保证有序性 防止指令重排

在不使用 volatile 的情况下，编译器和处理器可能会对指令进行优化重排序，这种重排序可能会改变代码中操作的执行顺序，但是在使用 volatile 修饰的变量时，编译器和处理器会禁止对 volatile 变量相关的操作进行重排序。

具体来说，volatile 关键字确保了：

编译器不会对 volatile 变量相关的代码进行重排序：编译器会在 volatile 变量的读写操作前后插入内存屏障（Memory Barrier），防止相关的指令重排序。

处理器不会对 volatile 变量的读写操作进行重排序：处理器遵循内存屏障的规则，保证了在 volatile 变量的写入操作之后，后续的读取操作不会被重排序到写入操作之前执行。

这种行为确保了 volatile 变量的写操作对后续的读操作具有立即可见性，并且防止了重排序可能引起的有序性问题。
在这个例子中，volatile 关键字确保了：

  1. 写线程对 counter 的增加操作在写回主存之后，不会发生重排序。
  2. 读线程在读取 counter 的值时，能够直接从主存中获取最新的值，而不会因为指令重排而读取到过期的值。
  3. 这种使用 volatile 的方式确保了操作的有序性和可见性，避免了潜在的指令重排可能引起的问题。
2.1.3.3 不保证线程安全

线程 1 读取 counter 的值为 0。
在线程 1 执行增加操作前，线程 2 也读取了 counter 的值为 0。
线程 1 增加后的值写回变量 counter，变为 1。
然后线程 2 也将它读取的值增加后写回，仍然是 1，而不是期望的 2。
这种情况下，两个线程对 volatile 变量的自增操作虽然使用了 volatile 修饰，但由于自增操作不是原子操作，可能会导致最终结果不符合预期

2.2 synchronized

同步（Synchronization）是指控制多个线程之间对共享资源的访问，以防止出现数据不一致或意外情况的一种机制。在多线程环境中，当多个线程并发访问共享资源时，同步机制确保了线程按照一定的顺序或规则来访问共享资源，避免了数据竞争和并发问题
